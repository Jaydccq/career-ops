# Jobright High Score Formal Reports

## Background

The local Jobright/newgrad scanner has already written high-scoring candidates
to `data/pipeline.md`, many with local JD caches under `jds/`. The user asked to
turn the previously local high-scoring Jobright list into formal reports.

## Goal

Generate formal evaluation reports for all currently pending high-scoring
Jobright candidates that are not already represented in existing reports or the
tracker.

## Scope

- Use the existing bridge `/v1/newgrad-scan/pending` reader as the source of
  pending candidates.
- Use `/v1/evaluate` with `evaluationMode: default` so each candidate attempts a
  full formal report rather than a quick screen.
- Run evaluations sequentially to reduce CLI, rate-limit, and tracker merge
  contention.
- Rebuild the dashboard after the batch.
- Do not submit applications or mark roles as applied.

## Assumptions

- "Jobright high score" means candidates still returned by the local pending
  reader after score, hard-filter, duplicate, tracker, and report-url filters.
- The local cached JD text is sufficient for formal report generation when
  present.
- Tracker merge may skip a valid report if `merge-tracker.mjs` considers it a
  lower-scored fuzzy duplicate of an existing tracker row.

## Implementation Steps

1. Start the real bridge.
   Verify: `/v1/health` returns real execution mode and repo checks pass.
2. Inventory pending candidates.
   Verify: save count, company, role, line number, local JD path, and value score
   in the progress log.
3. Evaluate candidates sequentially with `evaluationMode: default`.
   Verify: each job reaches `completed` or `failed`, and record report number or
   failure reason.
4. Rebuild dashboard.
   Verify: `npm run dashboard` completes.
5. Run targeted bridge verification.
   Verify: relevant server test still passes if bridge code changed.

## Verification Approach

- Bridge job snapshots for terminal state.
- Generated `reports/*.md` files.
- `data/applications.md` and merge summaries for tracker outcomes.
- `web/index.html` regenerated by `npm run dashboard`.

## Key Decisions

- Use formal default evaluation for this batch. `newgrad_quick` is useful for
  queue gating, but the user explicitly asked for formal reports.
- Process sequentially to make failures attributable and avoid merge lock
  contention.

## Risks and Blockers

- CLI/network instability can fail individual report jobs.
- Some candidates may be skipped by tracker merge deduplication even when the
  report is generated.
- The batch can be slow and may generate several ignored user-layer artifacts.

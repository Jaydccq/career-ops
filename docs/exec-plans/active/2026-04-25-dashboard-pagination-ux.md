# Dashboard Pagination UX

## Background

`bun run dashboard` serves the static dashboard from `web/template.html` with fresh repository data injected by `web/build-dashboard.mjs`. The current Reports, Tracker, Pipeline, and Scan History views render long filtered result sets in one continuous list or table, which makes reviewing and organizing large scan results difficult.

## Goal

Add pagination to the dashboard's long list views so filtered results can be reviewed in smaller, predictable batches.

## Scope

- Add shared client-side pagination behavior to `web/template.html`.
- Apply it to Reports, Tracker, Pipeline, and Scan History.
- Preserve current filters, sorting, table headers, action buttons, and static dashboard behavior.
- Rebuild `web/index.html` from the template.

Out of scope:
- Server-side pagination.
- Changes to scanner, tracker, reports, or application data semantics.
- Reworking the dashboard visual identity beyond controls needed for pagination.

## Assumptions

- `web/template.html` is the source of truth for the dashboard UI.
- `web/index.html` may be regenerated by `npm run dashboard:build`.
- Client-side pagination is sufficient for the current static dashboard data size.
- Page state can reset to the first page when filters change, which is more predictable than leaving the user on an empty later page.

## Implementation Steps

1. Add reusable pagination state and rendering helpers.
   Verify: Helpers clamp invalid pages and expose a stable page slice.
2. Add pagination controls to long dashboard sections.
   Verify: Reports, Tracker, Pipeline, and Scan History each show range, page size, previous, next, and page count.
3. Update render functions to paginate after filtering and sorting.
   Verify: Filter and sort changes reset to page 1; action buttons still bind against visible rows.
4. Rebuild the static dashboard.
   Verify: `npm run dashboard:build` succeeds and writes `web/index.html`.
5. Run a targeted behavior check.
   Verify: Generated HTML contains pagination controls and client-side render code can execute without syntax errors.

## Verification Approach

- Run `npm run dashboard:build`.
- Run a targeted Node/DOM smoke check if available without installing new dependencies.
- Prefer browser verification if the local dashboard can be opened through the available browser tooling.

## Key Decisions

- Use client-side pagination because the dashboard is a static HTML artifact and already holds data in `window.DATA`.
- Keep the existing editorial console style and add utilitarian controls that match current filters.
- Reset paginated views to page 1 after filters, sort changes, table header sort, toggles, or reset actions so users do not land on an empty later page.

## Risks and Blockers

- `web/index.html` is already modified before this task; rebuilding it may include pre-existing generated data changes.
- Browser-level smoke verification was available through headless Playwright.
